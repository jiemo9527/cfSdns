# AGENTS.md

## 1. 目标

这份文档给后续在本仓库内执行重构、修 bug、补验证逻辑的工程代理/开发者使用。

项目当前不是典型的分层应用，而是一个以 `src/main.py` 为中心的单进程轮询脚本：

1. 从多个外部数据源抓取 Cloudflare 候选 IP。
2. 把候选 IP 写入阿里云 DNS 的临时子域名。
3. 借助 Playwright 自动打开 IT-Dog 做第一轮测速。
4. 从测速结果中按运营商筛选优质 IP。
5. 把筛选后的 IP 写入正式子域名。
6. 再做一次测速，删除明显异常的 DNS 记录。
7. 休眠一段时间后进入下一轮。

核心特征：

- 强依赖第三方站点结构和返回格式。
- 强依赖外部网络环境与浏览器自动化稳定性。
- 没有自动化测试，当前验证方式主要是脚本级联调。
- 当前最重要的工作不是“加功能”，而是提升可维护性、可观测性和失败时的可诊断性。

## 2. 仓库结构

根目录：

- `README.md`：用户级运行说明，内容能帮助理解背景，但与当前代码并非完全一致。
- `requirements.txt`：Python 依赖。
- `Dockerfile`：基于 Playwright 官方 Python 镜像构建容器。
- `docker-compose.yml`：容器启动示例。
- `src/`：所有业务代码都在这里。

`src/` 目录：

- `main.py`：唯一真实主流程入口。
- `getIPFromW3.py`：候选 IP 聚合器，负责调用多个来源并做去重/过滤/抽样。
- `getv3data.py`：一个单独数据源的逆向请求逻辑，包含加密 token 生成和响应解密。
- `webTestUnion.py`：Playwright 自动测速逻辑，当前生产流程只使用 IT-Dog 分支。
- `cf2alidns.py`：阿里云 DNS 查询、添加、淘汰最旧记录、按值删除记录。
- `project_constants.py`：集中定义运行常量和阈值。
- `project_config.py`：集中处理 `.env` 加载和运行配置。
- `workflow_rules.py`：集中放置与主流程筛选/剔除相关的纯规则函数。
- `.env`：仓库内现存环境变量样例，但其位置与 README 描述不完全一致。

运行时副产物：

- `src/itdog_userdata/`：IT-Dog Playwright 持久化用户数据目录，运行后会生成。
- `src/cesu_userdata/`：CESU.AI 持久化用户数据目录，当前主流程未使用。
- `linux_error.png`、`cesu_error.png`：测速失败时可能生成的截图。
- `src/__pycache__/`：Python 缓存目录。

## 3. 真实入口与启动方式

当前真实入口是：`src/main.py`

标准本地启动方式：

```bash
python -m src.main
```

兼容性说明：

- `python src/main.py` 仍可运行，但只作为兼容入口保留，不作为标准启动方式。
- `Dockerfile` 中的 `CMD ["python", "main.py"]` 仍是待修正的旧入口。

按当前仓库结构，根目录不存在 `main.py`，因此：

- `python main.py` 是错误用法。
- 本地开发和联调一律以 `python -m src.main` 为准。

## 4. 模块职责与调用链

### 4.1 主流程

`src/main.py` 负责调度全部子模块，实际流程如下：

1. 读取环境变量：`domain_rr`、`domain_root`、`SLEEPTIME`。
2. 固定使用临时子域名前缀 `temp` 作为第一次测速入口。
3. 调用 `getIPFromW3.get_cf_ips()` 获取三大运营商候选 IP。
4. 把候选 IP 以 `{mobile, unicom, telecom}` 字典形式写入临时子域名。
5. 调用 `webTestUnion.run_itdog_test()` 对 `temp.<domain_root>` 做测速。
6. 调用 `filter_and_select_ips()` 从测速结果中筛选优质 IP。
7. 把筛选后的 IP 写入正式子域名 `domain_rr.domain_root`。
8. 再次调用 `webTestUnion.run_itdog_test()` 对正式域名测速。
9. 根据第二次测速结果调用 `cf2alidns.delete_record_by_value()` 删除异常记录。
10. 记录日志，休眠 `SLEEPTIME` 秒，然后进入下一轮。

### 4.2 IP 来源聚合

`src/getIPFromW3.py` 的职责：

- 从多个站点抓取 Cloudflare 候选 IP。
- 对不同运营商按不同丢包阈值筛选。
- 对所有来源合并、去重。
- 剔除 `172.65.*.*` 网段。
- 每个运营商随机抽样最多 19 个 IP 返回。

当前已接入的数据源：

- `vps789.com`：通过 `getv3data.v3data()` 调用。
- `api.uouin.com/cloudflare.html`
- `www.wetest.vip/page/cloudflare/address_v4.html`
- `cf.090227.xyz`
- `ip.164746.xyz/ipTop10.html`

注意：

- 注释中已明确 `cf.090227.xyz` “待修复”，说明作者自己也认为该来源不稳定。
- 源站解析逻辑有 HTML 表格版、纯文本版、API 版三套实现，但主流程只真正使用了其中部分路径。

### 4.3 单站逆向数据源

`src/getv3data.py` 的职责：

- 为 `https://vps789.com/public/cfMonitorList` 构造加密 token。
- 用 DES-CBC 加密请求时间戳，并用另一组 key 解密响应 `message`。
- 从解密后的 `content` 中按运营商丢包率阈值分类 IP。

该模块是主流程中“最不透明”的一段代码，后续如需重构：

- 先保行为一致。
- 再把“请求、解密、筛选”拆层。
- 任何对 key、IV、返回结构假设的修改都必须谨慎。

### 4.4 浏览器测速

`src/webTestUnion.py` 当前包含两套测速逻辑：

- `run_itdog_test()`：主流程使用。
- `run_cesu_test()`：存在但主流程未调用。

`run_itdog_test()` 的关键行为：

- 使用 Playwright 持久化上下文启动 Chromium。
- 访问 `https://www.itdog.cn/http/`。
- 在页面中输入目标域名。
- 打开“高级选项”，强制勾选自定义 DNS。
- 固定把 DNS 服务器设置成 `119.29.29.29`。
- 点击“快速测试”。
- 依赖 WebSocket 消息中 `type == "finished"` 判断测试完成。
- 直接从页面表格 `#simpletable` 中提取结果，输出 JSON 字符串。

`run_cesu_test()` 的关键行为：

- 面向 `https://www.cesu.ai/http_batch` 的批量测速。
- 依赖不同的 WebSocket 事件与不同的表格结构。
- 当前可视为实验性辅助工具，而不是正式生产路径。

### 4.5 DNS 写入与删除

`src/cf2alidns.py` 的职责：

- 初始化阿里云 SDK 客户端。
- 查询全量记录或指定 RR 记录。
- 判断某条记录是否已存在。
- 在达到套餐上限时删除最旧记录。
- 新增 A 记录。
- 批量把 `{mobile, unicom, telecom}` 写入指定子域名。
- 根据 `RR + IP + Line` 删除坏记录。

主流程最依赖的函数：

- `update_aliyun_dns_records()`
- `delete_record_by_value()`

## 5. 配置与环境变量

代码中实际读取的环境变量：

- `ALIYUN_ACCESS_KEY_ID`
- `ALIYUN_ACCESS_KEY_SECRET`
- `ALIYUN_PACKAGE_NUM`
- `SLEEPTIME`
- `domain_rr`
- `domain_root`

语义说明：

- `ALIYUN_ACCESS_KEY_ID` / `ALIYUN_ACCESS_KEY_SECRET`：阿里云 DNS API 凭据。
- `ALIYUN_PACKAGE_NUM`：每条线路允许保留的记录上限，代码默认 `100`。
- `SLEEPTIME`：主循环每轮之间的休眠秒数。
- `domain_rr`：正式业务子域名前缀，比如 `www`。
- `domain_root`：主域名，比如 `example.com`。

当前配置上的不一致：

- 标准位置是仓库根目录 `.env`。
- 为兼容历史用法，代码仍会读取 `src/.env`。
- 如果两处都存在，以根目录 `.env` 为准。

这意味着后续排查配置问题时，要先确认：

- 是否按照标准方式从仓库根目录执行 `python -m src.main`。
- 当前生效的是根目录 `.env` 还是历史 `src/.env`。
- 容器内工作目录与本地运行目录是否一致。

## 6. 当前业务规则与硬编码阈值

后续重构时必须先识别这些规则，不要在未确认业务意图前擅自改动。

### 6.1 候选 IP 侧

- `getIPFromW3.py` 会剔除 `172.65.*.*` 网段。
- 每个运营商最终最多保留 19 个候选 IP。
- `getv3data.py` 和 `getIPFromW3.py` 中，不同运营商使用不同丢包率阈值。

### 6.2 第一次测速筛选

`main.py -> filter_and_select_ips()` 的规则：

- 只接受 `状态 == "530"` 的记录。
- 只接受 `总耗时 < 1.0s` 的记录。
- 根据 `检测点` 前缀把结果归类到移动、联通、电信。
- 每个运营商从去重后的结果里随机抽样最多 5 个。

这里有明显语义风险：

- 代码把 HTTP 状态 `530` 当作合格条件。
- 这可能是基于 Cloudflare 特定行为的经验规则，也可能是历史遗留误判。
- 在没有真实样本之前，不要轻易把它改成 `200` 或其他状态码。

### 6.3 第二次测速后的剔除规则

主流程第二轮测速后的删除条件：

- 若 `状态 == "失败"`，标记删除。
- 若 `总耗时 >= 8.0s`，标记删除。
- 删除时会结合 `检测点` 前缀映射到 `mobile/unicom/telecom`。
- 如果待删除记录超过 10 条，只处理前 5 条。

这几个数字也是硬编码规则，不是平台能力边界。

## 7. 外部依赖与脆弱点

这个项目的稳定性高度依赖第三方页面和网络状态，下面这些位置是后续重构/修 bug 时的重点高危区。

### 7.1 页面抓取脆弱

受影响文件：`getIPFromW3.py`

风险点：

- HTML 抓取依赖表格结构固定。
- 文本抓取依赖站点持续输出逗号分隔 IP。
- 任何源站加防护、改 DOM、改字段名，都会直接导致候选池缩水。

### 7.2 浏览器自动化脆弱

受影响文件：`webTestUnion.py`

风险点：

- 依赖中文按钮文案，例如“高级选项”“快速测试”。
- 依赖 WebSocket 消息格式不变。
- 依赖表格选择器 `#simpletable`。
- 依赖 anti-bot 策略未升级到直接封锁 Playwright。

### 7.3 DNS 操作容错不足

受影响文件：`cf2alidns.py`

风险点：

- `add_record()` 的异常处理里直接 `pass`，会吞掉真实失败原因。
- 一旦阿里云侧限流、线路名不兼容、参数不合法，日志可能不足以定位。
- 当前模块内既有旧的增删逻辑，也有新的批量更新逻辑，存在行为重复和维护成本。

### 7.4 启动与路径不一致

受影响文件：`README.md`、`Dockerfile`、`src/main.py`

风险点：

- 文档写法、容器入口和实际文件布局不一致。
- `.env` 实际放置位置不统一。
- 非包结构导入依赖“从 `src` 直接运行脚本”的上下文，后续一旦改成包化结构，导入会先炸。

## 8. 已知代码味道与重构优先级

建议优先级从高到低：

### P0

- 统一真实启动方式，修正 `README.md`、`Dockerfile`、`docker-compose.yml` 与实际入口不一致的问题。
- 统一 `.env` 查找策略，避免“本地能跑、容器里不能跑”的路径问题。
- 让 DNS 写入失败可见，去掉无声吞异常。

### P1

- 拆分 `main.py` 里的业务流程与规则判断，把“采集、测速、筛选、同步、剔除”拆成可单测的函数。
- 把所有硬编码阈值抽为集中配置常量。
- 为外部站点访问增加更清晰的失败分类和日志上下文。

### P2

- 明确 `run_cesu_test()` 是否保留；不用就移出主模块或改成实验工具。
- 逐步从脚本式目录过渡为包结构，例如 `src/<package>/...` 或至少引入更明确的模块边界。

## 10. 改动时的工作原则

### 10.1 先保行为，再做结构优化

这是一个集成型脚本，不是纯算法仓库。很多看起来“不优雅”的逻辑，可能其实是为了兼容某个第三方站点的现实表现。

所以重构顺序应该是：

1. 先把当前行为摸清楚。
2. 先补日志、补小粒度函数、补可观察性。
3. 再做结构拆分。
4. 最后再考虑替换规则或调整阈值。

### 10.2 对第三方页面选择器保持保守

如果要改 `webTestUnion.py`：

- 优先增加兜底与日志，不要一上来就大改整段页面操作。
- 记录失败时卡在哪一步：打开页面、输入域名、展开高级项、点击测试、等待 WebSocket、提取表格。
- 选择器修改要尽量保留旧逻辑作为对照，至少在一次变更中不要同时改等待策略和 DOM 提取策略。

### 10.3 对阿里云线路字段保持谨慎

代码里使用的线路名是：

- `mobile`
- `unicom`
- `telecom`

但 SDK/控制台侧的线路行为是否完全一致，要以实际 API 返回为准。凡是涉及线路映射的改动，都要先检查 `query_all_domain_records()` 拿回来的 `Line` 字段原始值。

### 10.4 优先补错误上下文

当前项目里不少地方是 `print()` 或宽泛 `except Exception`。后续修 bug 时，优先补充：

- 正在访问哪个 URL
- 正在操作哪个域名/子域名
- 哪个运营商线路出错
- 哪个阶段出错（采集/测速/同步/删除）
- 原始异常内容

## 11. 验证建议

项目当前没有自动化测试，所以任何改动都应至少做下面一种验证。

### 11.1 纯代码级验证

适用于不方便访问外网时：

- 确认导入链没有被改坏。
- 确认函数签名和返回值形状没有意外变化。
- 确认配置读取路径更清晰而不是更混乱。

### 11.2 模块级联调

可分别验证：

- 单独跑 `getIPFromW3.py` 看是否还能拿到候选 IP。
- 单独跑 `webTestUnion.py` 看 Playwright 页面流程是否还能完成。
- 单独对 `cf2alidns.py` 的查询/删除函数做小范围验证，不要一上来就批量写正式域名。

### 11.3 全链路验证

最接近真实运行，但风险也最高：

- 使用测试域名而不是生产域名。
- 优先验证 `temp` 临时子域名流程。
- 验证新增记录、测速结果、坏记录删除三段链路是否闭环。

## 12. 当前缺失项

后续如要把项目从“可跑”提升到“可维护”，建议逐步补齐：

- 自动化测试，哪怕先从纯函数和解析函数开始。
- 明确的日志标准，替代 `print` 和静默异常。
- 统一配置模块。
- 统一常量模块。
- 对外部站点的适配层抽象。
- 一个稳定的包结构或至少清晰的运行目录约定。

## 13. 一句话判断项目现状

这是一个“业务路径明确，但实现高度依赖外部站点且缺少防护”的集成型脚本项目。

后续所有重构和 bug 修复，都应围绕三件事展开：

1. 降低外部依赖变化带来的脆弱性。
2. 提高失败时的可观测性。
3. 在不破坏现有筛选规则的前提下，把主流程拆得更容易验证。
